This NiFi workflow ingests raw streaming data from Kafka, enriches each event with additional metadata, routes events by type, and publishes the cleaned/enriched data into three typed Kafka topics. Each event type is also archived to disk for verification and debugging.

# Flow Design
### 1. Processor: ```ConsumeKafka```
- Topic: ```ds551-f25.team07.raw```
- Purpose: Continuosly consumes untyped JSON events generated by the DS551 event generator (from kafka). After it does that, it emits each event as a FlowFile for downstream processing 

### 2. Processor: ```EvaluateJsonPath```
- Using the ExtractText to ```event_type``` from the raw json.
- Stores the ```event_type``` into the flowfile attribute which will enable routing decision downstream.
- Here we terminated failed relationships and unmatched relationships as the main focus were the 3 event types.

### 3. Processor: ```RouteOnAttribute```
- Here we used the extracted ```event_type``` attribute to send each FlowFile down the correct branch.
- Example
Property: ```symptom_report``` Value: ```${event_type:equals("clinic_visit")}```
- Provides the data to have type-specific paths.
- Again, we would terminate unmatched relationships to avoid long processing time

### 4. Processor: ```UpdateRecord```
- Starting with enriching ```symptom_reports``` we included various metadata.
```
  1. "processing_timestamp" : "2025-12-11 00:30:12""
  2. "event_source" : "ds551-event-generator"
  3. "team_id" : "07"
```

- For the 2 other event types, it is the exact same process. Every event type had to have the same metadata.
- Lastly, again we automatically terminated any failures.

### 5. Processor: ```PublishKafka```
- For symptom_reports in this case, it writes and publishes the final enriched events to ```ds551-f25.team07.symptom_reports```

- Again we terminated any failures in the relationship.

### 6. Processor: ```PutFile```
- Every branch terminates with a PutFile processor that saves the final enriched JSON events into team-specific folders.
- This provides logs for proof and debugging support.

## Challenges
- One major challenge was that Apache NiFi was completely new to us, and initially we did not understand how data flows between processors. To overcome this, we used the NiFi documentation provided by the professor along with generative AI tools. ChatGPT helped us understand the overall workflow, processor relationships, and how record-based processors operate, which made it easier to design the flow.

- We also faced several processor configuration issues, where processors remained invalid due to missing or incorrect settings. By inspecting validation errors and testing configurations, we learned how to properly configure required properties before connecting processors.
- Our Kafka Consumer processor was not configured correctly. With guidance from the TA, we learned how to set the correct bootstrap servers, topic names, and group IDs, which allowed the flow to consume messages reliably.
